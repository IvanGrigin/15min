# AGENTS.md

Этот файл задает обязательные правила для всех будущих агентов, которые работают в этом проекте.

## Цель

Нужно, чтобы любой следующий агент:

- быстро понимал структуру проекта;
- не тратил лишние токены на повторный поиск;
- поддерживал единый порядок размещения файлов;
- обновлял документацию после изменений;
- фиксировал, что именно он сделал и где теперь лежит логика.


## Обязательная структура проекта

Агент обязан поддерживать такую схему:

- `problemgen/` — переиспользуемая логика генерации;
- `scripts/` — только запускаемые вручную скрипты и entry point-файлы;
- `outputs/` — сгенерированные артефакты;
- `docs/` — служебная документация и журнал;
- `tests/` — автоматические проверки.

Дополнительные правила:

- не складывать новые генераторы в корень проекта;
- не создавать новые безымянные файлы вида `15.py`, `28.py`, `final.py`, `new_version.py`;
- для новых скриптов использовать смысловые имена, например `generate_ratio_tasks.py`;
- если логика может переиспользоваться, она должна жить в `problemgen/`, а не в `scripts/`;
- если файл является результатом работы программы, он должен лежать в `outputs/`, а не рядом с кодом.


## Обязательные действия перед началом работы

Перед любыми изменениями агент обязан прочитать:

1. `DOCUMENTATION.md`
2. `docs/README.md`
3. `docs/FILE_INDEX.md`
4. `docs/VECTOR_TREE.md`
5. последние записи в `docs/WORK_LOG.md`

Если задача относится к конкретному файлу, агент должен сначала открыть именно этот файл, а не искать вслепую по всему проекту.


## Обязательные действия после завершения работы

После любых изменений агент обязан:

1. Обновить `docs/WORK_LOG.md`
2. При необходимости обновить `docs/FILE_INDEX.md`
3. При необходимости обновить `docs/VECTOR_TREE.md`
4. Если добавлен новый файл:
   - вписать его в индекс файлов;
   - кратко объяснить, зачем он нужен;
   - указать, кто его использует
5. Если изменена логика:
   - указать, в каком файле лежит новая логика;
   - указать, какие старые разделы проекта она затрагивает


## Что писать в журнале

В `docs/WORK_LOG.md` каждая новая запись должна содержать:

- дату;
- краткое имя задачи;
- что было сделано;
- какие файлы изменены;
- какие новые файлы созданы;
- что нужно знать следующему агенту;
- были ли проверки, тесты или ручной прогон.


## Как описывать файлы

Если агент меняет или добавляет файл, он должен указать:

- путь к файлу;
- назначение файла;
- ключевые сущности внутри файла;
- связи с другими файлами.

Это делается в `docs/FILE_INDEX.md`.


## Как поддерживать смысловое дерево

В `docs/VECTOR_TREE.md` должен поддерживаться компактный семантический индекс.

Идея:

- не хранить длинные пересказы;
- хранить маршруты вида:
  - если нужен верхний запуск новой архитектуры -> смотреть `scripts/run_problemgen.py` -> `main`
  - если нужен сюжет дружбы в классе -> смотреть `scripts/generate_friendship_class.py` -> `build_problem`, `main`
  - если нужен веб-интерфейс -> смотреть `problemgen/web/server.py`

Это дерево должно помогать следующему агенту быстро попасть в нужное место кода.


## Правила качества

Любой агент должен:

- сохранять согласованность русского языка;
- проверять, что сюжет соответствует масштабу чисел и единиц измерения;
- не оставлять за собой немые изменения без документации;
- не дублировать существующие документы без необходимости;
- не создавать новый источник истины, если уже есть подходящий файл в `docs/`;
- при добавлении нового генератора сразу решать, что является кодом, а что результатом генерации.


## Единый источник служебной документации

Для проектной навигации использовать:

- `DOCUMENTATION.md` — подробная техническая документация;
- `docs/FILE_INDEX.md` — индекс файлов;
- `docs/VECTOR_TREE.md` — короткое дерево смысловой навигации;
- `docs/WORK_LOG.md` — журнал выполненных работ.


## Если агенту нужно добавить новый раздел проекта

Тогда он обязан:

1. добавить код;
2. положить entry point в `scripts/`, если нужен отдельный запуск;
3. определить папку в `outputs/` для результатов, если генератор что-то сохраняет;
4. обновить индекс файлов;
5. обновить смысловое дерево;
6. оставить запись в журнале;
7. указать, как новый раздел искать и использовать.


## Формат приоритета

Приоритет документов такой:

1. `AGENTS.md`
2. `DOCUMENTATION.md`
3. `docs/VECTOR_TREE.md`
4. `docs/FILE_INDEX.md`
5. `docs/WORK_LOG.md`


## Короткая инструкция для следующего агента

Если времени мало, сначала открой:

1. `AGENTS.md`
2. `docs/VECTOR_TREE.md`
3. нужный файл из `docs/FILE_INDEX.md`

И только потом начинай редактирование.
