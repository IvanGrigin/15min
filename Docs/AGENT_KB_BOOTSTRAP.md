# Иерархическая база знаний для агентов: инструкция тиражирования

Документ самодостаточен: по нему агент внедряет базу знаний (БЗ) на
[TriliumNext Trilium](https://github.com/TriliumNext/trilium) в **любом**
проекте, независимо от стека. Референсная реализация — этот репозиторий:
`scripts/kb_server.mjs`, `scripts/kb.mjs`, `knowledge/`, `Docs/KNOWLEDGE_BASE.md`.

## Суть подхода

- Trilium — иерархические заметки: дерево произвольной глубины, полнотекстовый
  поиск, веб-интерфейс для людей и REST **ETAPI** для агентов.
- Запуск **локально без docker** и без сторонних серверов: checkout
  github.com/TriliumNext/trilium вне репозитория проекта,
  `pnpm install --frozen-lockfile && pnpm run --filter server build`, затем
  `node apps/server/dist/main.cjs` с `TRILIUM_DATA_DIR` и config.ini
  (`[General] noAuthentication=true`, `[Network] host=127.0.0.1`, свой порт на
  проект). Свежая БД инициализируется `POST /api/setup/new-document`.
  noAuthentication допустим только для loopback. Один checkout можно разделять
  между проектами — каталог данных у каждого проекта свой.
- Совместная работа разработчиков — через git репозитория проекта: markdown-
  зеркало дерева (заметка с детьми → каталог + `_note.md`, лист → `<title>.md`),
  команды CLI `sync-out`/`sync-in`; у каждого разработчика свой локальный
  Trilium, истина — каталог зеркала в git. `sync-in` ничего не удаляет; журнал
  append-only, конфликт слияния решается сохранением обеих строк.
- Git всегда остаётся источником истины для кода; БЗ — накопительная память
  между агентами, экономящая токены на восстановление контекста: короткие
  карты-маршруты «нужно X → открой файл Y», карточки подсистем, разборы
  регрессий (симптом → причина → фикс → защита), append-only журнал.
- Агенты работают с БЗ через тонкий CLI без зависимостей (Node fetch → ETAPI):
  адресация заметок человекочитаемым путём `"Раздел / Подраздел / Заметка"`.
- Ключевые ETAPI-вызовы: `GET /etapi/app-info`; `POST /etapi/create-note`
  `{parentNoteId, title, type:'code', mime:'text/markdown', content}` (корень
  дерева — `root`); `GET/PUT /etapi/notes/{id}/content`;
  `GET /etapi/notes?search=...`; у заметки есть `childNoteIds` — путь
  разрешается спуском по детям со сравнением `title`. Аутентификация —
  заголовок `Authorization: <ETAPI token>` (для loopback c noAuthentication —
  любая заглушка).

## Промт для агента

Скопируйте блок целиком, заполнив `<...>`:

```text
Внедри иерархическую базу знаний для агентов в проекте <ИМЯ ПРОЕКТА>
(рабочая директория <ПУТЬ>). База — TriliumNext Trilium
(github.com/TriliumNext/trilium), доступ для агентов через REST ETAPI.
Git остаётся источником истины для кода; база — накопительная память:
конспекты решений с причинами, разборы регрессий, журнал событий, навигация,
экономящая токены будущих агентов.

Перед началом: сделай git pull, изучи, как в проекте устроены основные
подсистемы и правила для агентов (CONTRIBUTING/AGENTS/Docs), чтобы разделы
БЗ были короткими картами-маршрутами со ссылками на файлы, а не пересказом.

Сделай ровно это:

1. Запуск локально, БЕЗ сторонних серверов и БЕЗ docker: клонируй
   github.com/TriliumNext/trilium ВНЕ репозитория проекта (например,
   ../tools/trilium), `pnpm install --frozen-lockfile`,
   `pnpm run --filter server build`; данные — отдельный каталог
   ../tools/trilium-data-<проект> с config.ini
   (`[General] noAuthentication=true`, `[Network] host=127.0.0.1`, свободный
   порт, у каждого проекта свой); лаунчер-скрипт в проекте запускает
   `node <checkout>/apps/server/dist/main.cjs` с TRILIUM_DATA_DIR, сам создаёт
   каталог данных с config.ini и инициализирует свежую БД
   `POST /api/setup/new-document`. Наружу порт не публиковать.

2. CLI для агентов (скрипт на Node >= 18 без зависимостей, глобальный fetch):
   команды info, ensure <путь>, put <путь> (--stdin|--content), get <путь>,
   append <путь> <строка> (дописывает "- YYYY-MM-DD: строка"), tree [путь],
   search <запрос>, а также sync-out <путь> <dir> и sync-in <путь> <dir> —
   markdown-зеркало поддерева для git (заметка с детьми → каталог, собственный
   текст ветки → _note.md, лист → <title>.md; sync-in создаёт/обновляет и
   ничего не удаляет). Путь = сегменты через " / ", разрешение от root спуском
   по childNoteIds со сравнением title, недостающие сегменты создаются через
   POST /etapi/create-note (type 'code', mime 'text/markdown'). Токены не
   печатать и не логировать. Добавь отслеживаемый каталог зеркала knowledge/
   с workflow: git pull → sync-in; правки → sync-out → commit → push.

3. Документ-контракт БЗ в docs/: зачем БЗ при живом git (правило приоритета:
   при расхождении верен репозиторий); запуск; ОБЯЗАТЕЛЬНАЯ иерархия под этот
   проект по шаблону «Обзор / Карта проекта / Подсистемы / Регрессии /
   Эксплуатация / Журнал (append-only)» с адаптированными названиями; правила
   агентов: после каждой завершённой задачи обновить заметку раздела и
   дописать строку в Журнал, затем sync-out и коммит зеркала в ту же ветку,
   что и код; каждый настоящий дефект — заметка «симптом → диагностика →
   причина → фикс → чем защищено» со ссылками на коммиты; заметки — краткий
   markdown со ссылками на файлы вместо копий кода; правила параллельной
   работы (журнал — только дописывание снизу, при конфликте сохранить обе
   строки; одну заметку в двух ветках не переписывать). Если в проекте уже
   есть обязательный журнал/индексы — БЗ их НЕ заменяет, в журнал БЗ пишется
   краткая строка со ссылкой. ЗАПРЕЩЕНО в БЗ: секреты, токены, персональные
   данные, длинные листинги кода.

4. Подними сервер, инициализируй БД, наполни иерархию реальным содержимым
   из изучения проекта (обзор, карта, карточки подсистем, известные
   регрессии из истории проекта, первая строка журнала), выполни sync-out и
   проверь раунд-трип: sync-out во временный каталог после sync-in обязан
   быть идентичен зеркалу (diff -r пустой). Покажи вывод info/tree/append.

5. Запрещено: коммитить токены и данные Trilium; вкапывать исходники Trilium
   в репозиторий (submodule/копия); публиковать порт наружу; класть в БЗ
   что-либо из списка запрещённого.

6. Заверши по правилам проекта: отдельная ветка, небольшие логические
   коммиты, запись в журнал изменений проекта, push и merge в основную ветку,
   чтобы БЗ появилась у остальных разработчиков после git pull.
```

## Критерии приёмки внедрения

1. Сервер поднимается одной командой на чистой машине (после разовой
   подготовки checkout) и слушает только loopback.
2. CLI: `info` печатает версию; `ensure` идемпотентен; `append` дописывает
   датированную строку; раунд-трип `sync-in` → `sync-out` даёт пустой `diff -r`.
3. Зеркало `knowledge/` в git содержит осмысленное первичное наполнение
   (не заглушки), контракт в docs/ — иерархию, workflow и запреты.
4. Ни в одном закоммиченном файле нет токена, пароля или данных Trilium.
5. Журнал изменений проекта дополнен; изменения влиты в основную ветку.
