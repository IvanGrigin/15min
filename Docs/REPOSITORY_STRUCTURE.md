# Repository Structure

Этот документ фиксирует, что должно лежать в каждой важной папке и как будущие агенты должны ориентироваться по репозиторию.

## Корень проекта

В корне должны лежать только:

- обязательные верхнеуровневые документы;
- лицензия;
- верхнеуровневые каталоги кода, данных, тестов и артефактов.

В корне не должны появляться:

- одноразовые генераторы;
- временные JSON;
- файлы с неясными именами;
- дубли служебной документации.


## `Docs/`

Что здесь должно быть:

- документация проекта;
- индексы и журнал;
- архитектурные blueprint-документы;
- `all_tasks_all_files.md` как неизменяемый корпус задач.

Что здесь не должно быть:

- runtime-данные;
- код генератора;
- новые результаты работы программы.

Примеры маршрутов:

- если нужна архитектура -> `Docs/ARCHITECTURE_BLUEPRINT.md`
- если нужна структура директорий -> `Docs/REPOSITORY_STRUCTURE.md`
- если нужен путь данных -> `Docs/DATA_FLOW.md`


## `data/`

Что здесь должно быть:

- JSON-схемы;
- классификаторы;
- каталоги шаблонов;
- словари сущностей;
- языковые словари;
- индекс исходных задач.

Что здесь не должно быть:

- бизнес-логика генерации;
- web-слой;
- результаты генерации.

Примеры будущих функций, которые будут читать этот слой:

- `load_task_template_schema()`
- `load_category_taxonomy()`
- `load_character_repository()`
- `load_verb_lexicon()`
- `load_source_task_index()`


## `problemgen/`

Что здесь должно быть:

- Python-код генератора;
- модели данных;
- загрузчики;
- генераторы;
- валидаторы;
- доменные адаптеры;
- веб и CLI.

Что здесь не должно быть:

- основной корпус задач;
- длинные списки строковых словарей, если они уже вынесены в `data/`;
- новые разовые скрипты.


## `problemgen/core/`

Должно содержать:

- общие модели;
- базовые типы результата;
- общие настройки и перечисления.

Связи:

- используется всеми другими пакетами;
- не должен зависеть от web-слоя.


## `problemgen/catalog/`

Должно содержать:

- загрузку и фильтрацию каталогов шаблонов;
- работу с индексом задач;
- поиск подходящих шаблонов.

Примеры будущих функций:

- `load_template_catalog()`
- `find_templates_by_category()`
- `find_templates_by_feature_flags()`
- `resolve_source_reference()`


## `problemgen/entities/`

Должно содержать:

- загрузку словарей персонажей, предметов и локаций;
- выбор сущности по роли;
- проверку совместимости.

Примеры будущих функций:

- `load_entity_repository()`
- `pick_entity_for_role()`
- `validate_role_entity_binding()`


## `problemgen/language/`

Должно содержать новый основной языковой слой.

Подслои:

- `morphology/` — словоформы, падежи, число, род, одушевленность;
- `syntax/` — управление, предлоги, глагольные рамки;
- `renderer/` — сборка финального текста;
- `validators/` — проверка грамматических и структурных ошибок.

Примеры будущих функций:

- `resolve_case_form()`
- `resolve_verb_form()`
- `render_task_text()`
- `validate_rendered_task()`


## `problemgen/russian/`

Это переходный совместимый слой.

Правило:

- новые сложные языковые механизмы нужно проектировать в `problemgen/language/`;
- в `problemgen/russian/` можно оставлять только совместимость и простые утилиты до полной миграции.


## `problemgen/generation/`

Должно содержать оркестрацию сборки экземпляра задачи.

Подслои:

- `selection/`
- `numbers/`
- `binding/`
- `assembly/`

Примеры будущих функций:

- `select_template()`
- `sample_numeric_parameters()`
- `bind_entities_to_slots()`
- `build_task_instance()`


## `problemgen/io/`

Должно содержать:

- чтение данных из `data/`;
- сохранение итоговых JSON;
- преобразование моделей в словари и обратно.

Примеры будущих функций:

- `read_json_file()`
- `write_task_bundle()`
- `dump_task_instance()`


## `problemgen/domains/`

Должно содержать:

- доменные адаптеры;
- тонкие настройки для семейств задач;
- регистрацию доменов.

Не должно содержать:

- огромные словари персонажей;
- общую морфологию;
- общий JSON-рендеринг.


## `problemgen/web/`

Должно содержать:

- локальный HTTP-сервер;
- thin adapter к каталогу и генератору;
- выдачу JSON и HTML.

Этот слой должен использовать готовые сервисы, а не дублировать их.


## `frontend/`

Должно содержать только клиентский UI:

- стили;
- JS для формы;
- легкую логику отображения.


## `scripts/`

Должно содержать только ручные точки запуска.

Правило:

- если в скрипте появляется много логики, ее надо выносить в `problemgen/`.


## `outputs/`

Должно содержать:

- примеры;
- массовые генерации;
- итоговые JSON и текстовые наборы.

Правило:

- новый runtime output писать только сюда;
- не создавать параллельную папку `output/` как отдельный источник артефактов.


## `tests/`

Должно содержать:

- `unit/` — проверки отдельных функций и небольших сервисов;
- `integration/` — проверка полного конвейера;
- `golden/` — эталонные задачи и тексты;
- `fixtures/` — тестовые входные данные.

Примеры сценариев:

- проверка падежной подстановки;
- проверка глагольного управления;
- проверка структуры JSON;
- проверка генерации без незаполненных слотов.
