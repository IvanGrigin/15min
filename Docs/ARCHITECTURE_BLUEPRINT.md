# Architecture Blueprint

Этот документ описывает целевую архитектуру проекта и обязателен для чтения перед структурными изменениями.

## Зачем нужен blueprint

Текущий проект уже умеет генерировать часть задач, но для полноценной системы ему нужно жесткое разделение между:

- read-only корпусом исходных задач;
- каталогом формализованных шаблонов;
- словарями сущностей и морфологии;
- генератором числовых параметров;
- рендерингом русского текста;
- валидацией;
- финальной JSON-выдачей.

Этот документ фиксирует именно такое разделение.


## Главная архитектурная идея

Проект должен работать не как набор Python-скриптов со случайными строками, а как конвейер:

1. исходный read-only корпус;
2. структурированная индексация;
3. абстрактные математические шаблоны;
4. сюжетные роли и сущности;
5. морфология и синтаксис;
6. генерация конкретного экземпляра;
7. языковой рендеринг;
8. валидация;
9. сохранение результата в JSON.


## Верхнеуровневые зоны репозитория

### `Docs/`

Здесь лежат:

- документация проекта;
- индекс файлов;
- навигационное дерево;
- журнал;
- read-only файл `all_tasks_all_files.md`.

Правило:

- `Docs/all_tasks_all_files.md` не менять;
- архитектурные документы здесь менять можно и нужно.


### `data/`

Это слой проектных данных, а не код.

Здесь должны жить:

- JSON-схемы и примерные структуры;
- таксономии категорий, жанров и feature flags;
- индекс задач из read-only корпуса;
- шаблоны задач;
- словари персонажей, локаций, предметов и ролей;
- словари глаголов, предлогов и правил согласования.


### `problemgen/`

Это слой исполняемой логики.

Здесь должны жить:

- модели;
- загрузчики данных;
- генератор чисел;
- связывание ролей с сущностями;
- языковой рендеринг;
- валидация;
- доменные адаптеры;
- выдача результата в CLI и web.


### `scripts/`

Только ручные точки запуска.

Они не должны быть местом, где живет реальная бизнес-логика.


### `outputs/`

Только артефакты генерации:

- сгенерированные JSON;
- текстовые выгрузки;
- примеры.


### `tests/`

Только автоматические проверки.

Отдельно должны быть:

- unit-тесты;
- integration-тесты;
- golden-наборы;
- fixtures.


## Целевые слои внутри `data/`

### `data/schemas/`

Содержит контракты и примеры структуры:

- шаблон задачи;
- экземпляр задачи;
- сущность;
- глагольная запись;
- при необходимости новые контракты.

Здесь не должно быть исполняемой логики.


### `data/taxonomy/`

Содержит классификаторы:

- суперкатегории;
- подкатегории;
- жанры;
- feature flags;
- шкалу сложности.


### `data/source_index/`

Содержит разметку задач из `Docs/all_tasks_all_files.md`:

- идентификаторы исходных задач;
- ссылки на разделы/блоки;
- группировку похожих задач;
- пометки о пригодности к параметризации.


### `data/templates/`

Содержит уже формализованные шаблоны задач.

Каждый шаблон должен описывать:

- математическую структуру;
- текстовый pattern со слотами;
- роли сущностей;
- ограничения на числа;
- ограничения на ответ;
- метки классификации;
- признаки фильтрации.


### `data/entities/`

Содержит словари:

- персонажей;
- локаций;
- предметов;
- ролей и совместимостей.

Сущности должны храниться не как сырые строки, а как нормализованные записи с грамматическими признаками и формами.


### `data/language/`

Содержит языковые словари и правила:

- глаголы;
- предлоги;
- служебные слова;
- правила согласования;
- при необходимости отдельные исключения.


## Целевые слои внутри `problemgen/`

### `problemgen/catalog/`

Этот слой работает с каталогом формализованных шаблонов и их поиском.

Он должен:

- читать `data/source_index/`;
- читать `data/templates/`;
- отдавать список доступных шаблонов по фильтрам;
- не заниматься языковым рендерингом.


### `problemgen/entities/`

Этот слой работает со словарями сущностей.

Он должен:

- загружать персонажей, локации и предметы;
- выбирать подходящие сущности по роли;
- проверять совместимость сущности и шаблона;
- возвращать структурированные данные, а не строки.


### `problemgen/language/`

Это новый основной языковой слой.

Он должен быть разделен хотя бы на:

- `morphology/`
- `syntax/`
- `renderer/`
- `validators/`

Текущий пакет `problemgen/russian/` следует считать переходным совместимым слоем.


### `problemgen/generation/`

Это слой сборки конкретного экземпляра задачи.

Он должен быть разделен хотя бы на:

- `selection/` — выбор шаблона;
- `numbers/` — генерация чисел;
- `binding/` — связывание ролей с сущностями и словоформами;
- `assembly/` — сборка полной структурированной задачи.


### `problemgen/io/`

Этот слой должен отвечать за:

- чтение JSON-данных;
- запись итоговых JSON;
- при необходимости сериализацию и десериализацию внутренних моделей.


### `problemgen/domains/`

Домены должны остаться, но стать тоньше.

Их задача:

- описывать семейства математических шаблонов;
- задавать доменные ограничения;
- указывать, какие шаблоны и фильтры доступны.

Домены не должны хранить в себе все языковые знания проекта.


## Что считается хорошей формулировкой задачи

Система должна быть построена так, чтобы задача:

- не содержала случайных сырой строк;
- не подставляла неподходящий падеж;
- не использовала глагол без учета рода и числа;
- не смешивала объект, персонажа и локацию в одной роли;
- могла быть объяснена по trace-данным, откуда взялась каждая словоформа.


## Обязательное правило для комментариев и README

Все архитектурные `README.md`, blueprint-документы и поясняющие комментарии в каркасных файлах считаются частью структуры проекта.

Нельзя:

- удалять их без замены;
- превращать архитектурный каталог в немую папку;
- оставлять новый слой без пояснения его назначения.


## Минимальный путь миграции

При постепенной перестройке проекта действовать так:

1. привести к единому стандарту документацию и пути артефактов;
2. ввести `data/` как отдельный слой;
3. вынести сущности и языковые словари из Python-констант в данные;
4. ввести новый пакет `problemgen/language/`;
5. ввести каталог шаблонов и индексацию исходных задач;
6. переводить домены на структурированные шаблоны по одному семейству за раз;
7. держать `legacy`-ветки изолированными до полной замены.
